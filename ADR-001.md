# ADR-001: Groq vs OpenRouter for LLM Inference

## Context
The AI Book Factory is required to operate at **zero API cost**. It needs a highly capable Large Language Model (comparable to GPT-4-class reasoning) to generate readable, engaging chapters (15k-50k words total) with narrative arc consistency and adherence to predefined outlines.  

Additionally, the generation process must be reasonably fast (targeting <30 minutes per book) but can tolerate queueing/batching.

The primary options evaluated were:
1. **OpenRouter (Free Tier Models)**
2. **Groq (Free Tier)**
3. **Pollinations.ai (Text API Fallback)**

## Decision
We elected to use **Groq Free Tier (Llama 3.3 70B Versatile)** as the primary inference engine, with an asynchronous `Semaphore`/Token Bucket limiter built in around the API rate limits. 

We utilize Pollinations.ai strictly as the **Image Generation fallback/primary** method due to its completely unrestricted, keyless Flux access. We do not use Pollinations Text for the primary generation because the context windows and parameters are harder to explicitly bound compared directly to an OpenAI-compatible API spec like `groq`.

## Consequences
Groq provides immense inference speed and access to a 70B model, meaning the prose output quality is consistently high enough to achieve a 60-70 Flesch-Kincaid readability score. 

**However, the primary trade-off is Groq's aggressive Free Tier limits (20 RPM).**
To mitigate this:
1. `book_factory.py` implements a pacing lock inside `AIWriter.generate()` (strict 3.1-second minimum delays between requests).
2. It wraps requests in robust `tenacity` retry structures configured for `HTTP 429` exponential backoffs.
3. Batch execution (`batch_generator.py`) is funneled through an SQLite queue to persist state so that generation can be safely paused and resumed if limits are blown unexpectedly.
